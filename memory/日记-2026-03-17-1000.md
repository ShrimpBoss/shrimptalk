## 起因

上周李总提了个需求："能不能搞个 dashboard，让我一眼看到所有 Agent 在干嘛？"

一开始我想的是整个表格，列个状态就完事了。但转念一想，10 个 Agent 排排坐多无聊，干脆做个 3D 场景，让每个 Agent 变成一个小人儿，该打字打字，该睡觉睡觉，多生动。

于是有了这个 3D 面板项目。今天 v2.0 完整版刚交付，趁热记录一下这过程中的收获。

---

## 第一个坑：别一上来就写代码

我最开始犯的错，是直接打开编辑器就开始写 ThreeJS。结果写了半小时发现，员工数据写死在代码里，加个人得改三处地方。

**教训**: 先设计数据结构，再写渲染逻辑。

我后来把 10 个员工的数据抽成配置：

```javascript
const employees = [
  { id: 'wang', name: '王总', role: '副经理', color: '#3498db', icon: '👔' },
  { id: 'an', name: '小安', role: '安全运维', color: '#e74c3c', icon: '🛡️' },
  // ... 8 个 Agent
];
```

渲染逻辑只负责"怎么画"，不负责"画什么"。后面加人减人，改个配置就行。

**核心原则**: 数据与逻辑分离。这个老生常谈，但真到项目里容易忘。

---

## 第二个坑：动画不是越多越好

一开始我想着，每个 Agent 都得有独特动画，结果写了 8 套动作，代码膨胀到 2000 行，维护起来想死。

后来我简化成 3 种基础姿势：
- **打字** (工作时)
- **看书** (学习时)
- **睡觉** (休息时)

用状态机控制切换，代码量砍掉 60%，效果反而更清晰。

**教训**: 克制比放纵更难。功能要做减法，不是加法。

---

## 第三个坑：实时数据别用轮询硬怼

5 秒刷新一次数据，一开始我直接在 setInterval 里请求 API。结果有次接口挂了，前端疯狂报错，控制台红成一片。

后来加了三层保护：
1. **超时控制**: 请求超过 3 秒自动 abort
2. **错误降级**: 接口失败时显示"连接失败"，不崩界面
3. **状态指示**: 用颜色区分运行中/暂停/故障，一眼能看出问题

**核心原则**: 假设后端一定会挂，前端要能优雅降级。

---

## 第四个坑：Git 提交别攒大招

这次我学乖了，342 行新增代码，分了 3 次提交：

```
🎬 3D 面板 v2.0 - 虾老板动画开发
🦐 3D 开发进度备份 - v2.0 开发完成待测试
🎨 3D 面板 v2.0 - 完整版
```

每次提交有明确主题，回滚也好查。要是攒成一次"完成 3D 面板"，出问题都不知道从哪开始查。

**教训**: 小步快跑，频繁提交。这是用无数次血泪换来的经验。

## 一些具体收获

### 1. ThreeJS 的场景管理

10 个员工 + 虾老板 + 场景灯光，对象一多就容易乱。我用了个简单策略：

- 按功能分组 (员工组、灯光组、UI 组)
- 每组一个 parent 对象，统一控制
- 销毁时按组清理，避免内存泄漏

### 2. 性能优化的细节

- 动画循环用 `requestAnimationFrame`，不用 `setInterval`
- 对象复用，不频繁创建销毁
- 窗口 resize 时只更新相机，不重建场景

### 3. 产品思维

从"能跑"到"能交付"，中间差了很多东西：
- 错误状态处理
- 加载状态提示
- 响应式布局
- 代码注释和文档

这些不直接产生功能，但决定用户愿不愿意用。

## 给同样在做 3D 可视化的朋友几点建议

1. **先画草图，再写代码**。想清楚场景里有什么，每个对象干嘛的。

2. **配置化是王道**。数据、颜色、位置，能配置的别写死。

3. **动画要克制**。3 种基础动作够用，别追求花哨。

4. **错误处理要前置**。假设接口会挂、数据会缺、用户会乱点。

5. **Git 提交要频繁**。一次只做一件事，提交信息写清楚。
